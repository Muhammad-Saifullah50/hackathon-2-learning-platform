# Research: AI Agent Layer (F07-F12)

## Decision 1: Orchestration Approach — Custom over OpenAI Agents SDK

**What was chosen**: Custom agent orchestration using existing `LlmClient`/`LlmService` with deterministic routing in a Triage Agent service class.

**Why chosen**:
- The codebase already has a fully functional LLM provider layer (F06) with streaming, caching, error handling
- Adding OpenAI Agents SDK would introduce a second LLM abstraction layer, creating confusion about which to use
- The spec's "handoff" concept maps cleanly to deterministic routing — the Triage Agent selects a specialist, and the specialist handles the request
- Keeps dependency surface minimal (no new pip packages beyond what's already in requirements.txt)
- Aligns with AGENTS.md anti-pattern: "Over-engineering early — no microservices, no message queues until users demand them"

**Alternatives considered**:
- OpenAI Agents SDK with native handoffs — would require adding `openai-agents` package and learning its API
- LangChain/LangGraph — heavyweight, overkill for 6 agents with simple routing
- Each agent as separate FastAPI microservice — violates AGENTS.md anti-pattern

## Decision 2: Deterministic Routing Implementation

**What was chosen**: Keyword/regex-based intent classification with confidence scoring.

**Why chosen**:
- FR-002 explicitly requires "deterministic code-based classification (not LLM-based)"
- Fast (< 50ms), testable, reproducible
- Regex patterns can match intent signals: "what is", "explain", "how does" → concept-explanation; "error", "TypeError", "why does my code fail" → code-debug; "review my code" → code-review; "exercise", "practice", "challenge" → exercise-generation; "progress", "how am I doing" → progress-summary
- Confidence scoring based on number of matched patterns and specificity

**Alternatives considered**:
- LLM-based intent classification — violates FR-002
- ML classifier (scikit-learn) — overkill, requires training data we don't have yet

## Decision 3: Conversation Session Persistence

**What was chosen**: SQLAlchemy model with JSONB column for conversation history.

**Why chosen**:
- Leverages existing Neon PostgreSQL (F02) and async session infrastructure
- JSONB allows flexible conversation history without schema changes
- Sessions scoped per tutoring interaction (not per day) — aligns with user expectation of "a conversation"
- Existing `Session` model in auth is for auth sessions, not conversations — separate model needed

**Alternatives considered**:
- Redis for session storage — would add new infrastructure dependency
- In-memory sessions — lost on restart, doesn't support multi-server deployment

## Decision 4: Progressive Hint State Management

**What was chosen**: `HintProgression` model in PostgreSQL tracking hint level (1-3), hints already provided, and solution reveal status.

**Why chosen**:
- Survives page reloads and session interruptions
- Enables analytics on hint effectiveness (do students resolve after hint 1, 2, or 3?)
- Simple state machine: level 1 → level 2 → level 3 → solution

**Alternatives considered**:
- Client-side hint state — lost on page reload, unreliable
- Embedding hint state in conversation history JSONB — harder to query for analytics

## Decision 5: Exercise Test Case Storage and Execution

**What was chosen**: Test cases stored as JSONB in Exercise model. Grading executes student code + test cases in DockerSandbox (F05).

**Why chosen**:
- JSONB allows flexible test case format (function name, inputs, expected outputs, or assert statements)
- DockerSandbox already implements isolation (F05 complete) — reuse it
- Test cases generated by Exercise Agent via LLM, validated before storage

**Alternatives considered**:
- Separate TestCase model — unnecessary normalization for JSON-structured data
- Running tests directly on server — violates security constraint (FR-041)

## Decision 6: Mastery Calculation with Incomplete Data

**What was chosen**: Proportional weight redistribution when components are missing.

**Why chosen**:
- FR-036 requires handling incomplete data gracefully
- If a student has exercises but no quizzes: exercises get 40/(40+20+10) = 57% of available weight, code quality 29%, streak 14%
- Response notes which components are missing
- Preserves the spirit of the formula without fabricating data

**Alternatives considered**:
- Default missing components to 0 — unfairly penalizes students who haven't had opportunity yet
- Require all components before calculating — blocks progress view entirely

## Decision 7: Single Unified Chat Endpoint

**What was chosen**: One `POST /api/v1/agents/chat` endpoint where Triage routes internally.

**Why chosen**:
- Students shouldn't need to pick which agent to talk to — they just ask questions
- Simpler frontend integration (one endpoint to call)
- Triage Agent handles routing transparently
- Individual agent endpoints still exist for programmatic access (exercise generation, progress summary)

**Alternatives considered**:
- Separate endpoint per agent — requires frontend to know which agent to call
- GraphQL with agent field — overkill for this use case
